# Phase 65 — Transaction Document UX

## الهدف
تحويل فاتورة البيع/الشراء من شكل Dialog ضيق إلى سطح عمل Transaction Document متناغم مع نظام التبويبات.

## المنجز
- إضافة `features.transactions.TransactionDocumentLayout` كعقد تخطيط موحد للمعاملات.
- إضافة `features.transactions.TransactionLineGrid` كجدول معاملات احترافي مشترك.
- ربط فاتورة البيع/الشراء بالـ `TransactionLineGrid` بدل ربطها مباشرة بجدول عام.
- حماية الأعمدة الأساسية من الإخفاء: الصنف، الكمية، الوحدة، الإجمالي.
- إبقاء جدول الفاتورة هو مساحة العمل الرئيسية بارتفاع أدنى أكبر وسلوك resize خاص.
- إخفاء بطاقة العنوان القديمة تلقائيًا عند تشغيل الفاتورة داخل Workspace Tab.
- إبقاء أزرار الحفظ/الطباعة/الإلغاء في شريط سفلي لا في أعلى النافذة.
- إضافة hidden imports خاصة بمكونات transactions لتفادي أخطاء PyInstaller.
- إضافة Guard جديد: `tools/transaction_document_ux_guard.py`.

## النتيجة العملية
الفاتورة أصبحت أقرب إلى قالب موحد يمكن استعماله لاحقًا في:
- فاتورة بيع
- فاتورة شراء
- مرتجع بيع
- مرتجع شراء
- POS
- Restaurant checkout

## ما لم يتغير عمدًا
لم يتم نقل منطق الحفظ والطباعة من `invoice_dialog.py` بالكامل في هذه المرحلة؛ تم تثبيت سطح التخطيط والجدول الموحد أولًا لتقليل مخاطر كسر الفوترة والطباعة.
