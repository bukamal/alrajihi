# Phase 294 — Restaurant Settings & Printer Routing Enforcement

## الهدف
تثبيت عقد إعدادات المطعم والطباعة بعد مراحل دورة الطلب، المطبخ، الدفع، الطباعة، المخزون، وعمليات الطاولات.

## ما تم تنفيذه
- إصلاح توافق Python قبل 3.12 في `kitchen_display_widget.py` عبر إزالة f-string المتداخلة التي كانت تفشل في `architecture_guard.py`.
- إضافة عقد إعدادات مطعم مركزي:
  - `alrajhi_client/features/restaurant/restaurant_settings_contract.py`
- تطبيع إعدادات المطعم:
  - ورق إيصال العميل.
  - ورق تذكرة المطبخ.
  - ورق ملخص الجلسة.
  - طابعة الإيصال.
  - طابعة المطبخ.
  - طابعة ملخص الجلسة.
  - الطباعة التلقائية لتذكرة المطبخ.
  - الطباعة التلقائية للإيصال بعد الإغلاق.
  - الطباعة التلقائية لملخص الجلسة بعد الإغلاق.
  - نسبة الخدمة الافتراضية.
  - نسبة الضريبة الافتراضية.
  - توقيت خصم المخزون: checkout أو served.
- توصيل `RestaurantPrintingBridge` بعقد التوجيه، بحيث يحمل كل payload معلومات `print_route` واضحة.
- تفعيل الطباعة التلقائية لملخص الجلسة بعد الإغلاق عند تشغيل الإعداد المقابل.
- توسيع تبويب إعدادات المطعم لإظهار حقول التوجيه والطباعة التلقائية.
- ربط المرحلة داخل Release Readiness Gate.

## القيود المقصودة
- لا تزال الطباعة المرئية موحدة على Browser HTML حسب قرارات مراحل الطباعة السابقة.
- أسماء الطابعات محفوظة كـ routing metadata للتشخيص أو للربط الخارجي، وليست مسار طباعة Qt مباشر.

## التحقق
- `python tools/architecture_guard.py`
- `python -m compileall -q alrajhi_client alrajhi_server tests tools`
- اختبارات Phase 294 واختبارات المطعم ذات الصلة.
- حراس release/packaging/runtime/phase32.
