# Phase 428 — Operational Item Card Grid Unification

## الهدف

توحيد عرض المواد داخل واجهة POS وواجهات المطعم/الكافي عبر شبكة بطاقات تشغيلية مشتركة، بثلاثة أعمدة افتراضيًا، بدون نسخ كود المطعم داخل POS وبدون لمس منطق البيع أو الدفع أو المخزون.

## ما تم

- إضافة مكوّن مشترك:
  `alrajhi_client/ui/operational_item_card_grid.py`
- المكوّن يعرض المواد كـ cards، افتراضيًا 3 أعمدة، مع fallback responsive إلى 2 أو 4 حسب عرض الشاشة.
- ربط POS بالمكوّن عبر `catalog_service.items()` فقط للعرض، ثم عند النقر يتم استخدام مسار POS الحالي `add_barcode_to_cart(..., mode='manual')`.
- ربط واجهة المطعم التشغيلية `RestaurantPOSWidget` بالمكوّن بدل بناء أزرار محلية داخل `QGridLayout`.
- ربط واجهة المطعم البسيطة `RestaurantSimplePOSWidget` بالمكوّن بدل العرض القديم صفًا واحدًا لكل مادة.
- إضافة QSS مشترك باسم `operationalItemCard*` حتى تستعمل POS والمطعم والكافي نفس السطح البصري.

## قواعد معمارية

- لا يوجد `QTableWidget` لعرض المواد.
- لا يوجد نسخ لكود بطاقات المطعم داخل POS.
- بطاقة المادة لا تعدّل المخزون أو الدفع أو الفاتورة بذاتها؛ هي تطلق إشارة اختيار فقط.
- Enter والجداول التحريرية تبقى تحت Phase426/427 ولا يتم المساس بها.
- POS يحتفظ بسرعة الباركود والدفع، والمطعم يحتفظ بسلوك الجلسات/المطبخ.

## الاختبار اليدوي

1. افتح POS.
2. تأكد أن المواد تظهر بثلاثة أعمدة تقريبًا.
3. اضغط على بطاقة مادة: يجب أن تضاف للسلة عبر منطق POS الحالي.
4. جرّب الباركود: يجب أن يبقى أسرع مسار، بدون تعارض مع البطاقات.
5. افتح واجهة المطعم: يجب أن تظهر المواد بنفس منطق البطاقات بثلاثة أعمدة.
6. غيّر اللغة RTL/LTR: يجب أن تبقى الشبكة متزنة.
