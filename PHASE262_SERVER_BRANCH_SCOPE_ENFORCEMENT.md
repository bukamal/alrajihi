# PHASE262 — Server-side Branch Scope Enforcement

## الهدف

تحويل عقد رقابة الفروع من Phase 261 من مستوى التعريف والـ helpers إلى إنفاذ فعلي داخل مسارات API الحساسة. هذه المرحلة لا تكتفي بأن يعرف العميل الفروع المسموحة، بل تجعل الخادم نفسه يفلتر ويمنع القراءة والتعديل والحذف خارج نطاق `user_branch_access`.

## ما تم

- تطبيق `branch_access_policy.scope_sql(...)` على قوائم الفواتير مع دعم `branch_id` في query string.
- فحص الفرع عند فتح فاتورة مفردة، تعديلها، حذفها، وتغيير سير العمل.
- فرض فرع فعال عند إنشاء الفاتورة إذا لم يرسل العميل فرعًا واضحًا.
- تطبيق نفس المبدأ على مرتجعات البيع والشراء: القوائم، فتح المرتجع، الفواتير القابلة للمرتجع، سطور المرتجع، الإنشاء، التعديل بالعكس/إعادة الإنشاء، والحذف.
- تطبيق نطاق الفروع على المستودعات: القوائم، الافتراضي، الكمية المتاحة، الأرصدة، الحركات، العكس، التحويلات، وإلغاء التحويل.
- تطبيق نطاق الفروع على الصناديق والحسابات البنكية: القوائم، الافتراضي، الفتح، الإنشاء، التعديل، والحذف.
- إدخال فلترة الفرع في ملخصات التقارير الأساسية مثل summary و income statement، مع الحفاظ على دعم الفلاتر التاريخية.
- إضافة helper جديد `alrajhi_server/services/branch_scoped_sql.py` ليكون نقطة توسع مركزية للتقارير والمسارات القادمة.

## ملاحظات معمارية

- إذا كان المستخدم admin أو لديه صلاحية `branches.view_all` يمكنه رؤية كل الفروع أو تحديد فرع معين عبر `branch_id`.
- إذا كان المستخدم مقيدًا بفروع محددة، يعيد `scope_sql` فلترة SQL تلقائيًا إلى الفروع المسموحة.
- إذا لم تكن بيانات `user_branch_access` مزروعة لمستخدم قديم، يبقى السلوك backward-compatible ولا يقفل كل البيانات فجأة.
- الرد الموحد عند منع الوصول هو:

```json
{"code":"BRANCH_ACCESS_DENIED"}
```

## نطاق المرحلة

هذه المرحلة ركزت على المسارات الأعلى أثرًا: invoices, returns, warehouses, cashboxes, reports الأساسية. المطعم و POS يملكان Operational Shell من Phase 258 ويحتاجان مرحلة لاحقة لإدخال فلترة الفروع داخل كل routes التشغيلية الخاصة بهما تفصيليًا.
