# Phase 50 — Offline UI Stability Guard

## الهدف
فحص وإصلاح المسارات المشابهة لمشكلة انهيار التطبيق عند توقف الخادم أثناء أو بعد حفظ عملية قابلة للمزامنة Offline.

## سبب المشكلة المتكررة
بعض شاشات الواجهة كانت تنفذ قراءات REST غير قابلة للإدخال في Offline Queue بعد الحفظ أو أثناء التحديث، مثل:

- تحديث قائمة الفواتير بعد حفظ فاتورة Offline.
- تحديث المرتجعات.
- تحديث السندات.
- تحديث العملاء/الموردين.
- تحديث المواد عبر BaseWidget.
- تحديث المستودعات والأرصدة والحركات والتحويلات.
- تحديث الصناديق والبنوك.
- فتح حوارات الفواتير/التصنيع مع الخادم متوقفاً.

## الإصلاحات

- إضافة `core/offline_guard.py` كأداة مركزية لاكتشاف أخطاء القراءة عند انقطاع الخادم.
- تحصين `BaseWidget.refresh()` حتى لا تسقط أي شاشة مبنية عليه عند فشل القراءة.
- تحصين شاشات:
  - العملاء.
  - الموردون.
  - المواد، عبر BaseWidget.
  - الفواتير، سابقاً + تعزيز الفحص.
  - المرتجعات.
  - السندات.
  - المستودعات.
  - التصنيع.
  - الصناديق والبنوك.
- تحصين حوارات:
  - الفاتورة عند تحميل الأطراف والمواد.
  - BOM.
  - أمر التصنيع.
- إضافة `tools/offline_widget_guard.py` لفحص وجود الحماية في أهم مسارات الواجهة.

## السلوك المتوقع الآن

عند توقف الخادم:

- العملية القابلة للمزامنة تدخل Offline Queue.
- أي تحديث قراءة لاحق لا يسقط التطبيق.
- تظهر رسالة تحذير فقط.
- تبقى البيانات الحالية في الجدول كما هي إلى حين عودة الاتصال.
