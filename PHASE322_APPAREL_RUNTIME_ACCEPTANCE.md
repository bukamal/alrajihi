# PHASE 322 — Apparel Runtime Acceptance

## الهدف
إغلاق المسار التشغيلي الأول للألبسة بعد مراحل المتغيرات، شاشة الألبسة، الربط بالفواتير، عمليات المستودع، والتقارير.

هذه المرحلة لا تنشئ محرك ألبسة مستقلًا، بل تثبت أن الألبسة تعمل فوق نفس محرك المواد والمستودعات والفواتير والتقارير، مع حفظ هوية اللون/المقاس في كل خطوة.

## سيناريو القبول

التسلسل المقبول للألبسة هو:

1. إنشاء مادة أصلية مثل: قميص رجالي.
2. إنشاء متغيرات اللون/المقاس دفعة واحدة.
3. شراء كمية لمتغير محدد.
4. مسح باركود المتغير في البيع.
5. حفظ فاتورة البيع وخصم كمية المتغير نفسه.
6. إرجاع نفس المتغير في المرتجع أو عكس الحركة.
7. تحويل نفس المتغير بين مستودعين.
8. جرد أو تسوية نفس المتغير.
9. مراجعة تقرير الألبسة حسب اللون/المقاس.

## ضمانات المرحلة

- الباركود الذي يعود إلى متغير يجب أن يرجع اللون والمقاس الصحيحين.
- سطر الفاتورة يجب أن يحمل `variant_id` وبيانات اللون/المقاس.
- حركة المستودع يجب أن تحمل نفس هوية المتغير.
- عكس الحركة أو إلغاء الفاتورة يجب ألا يتحول إلى حركة عامة على المادة.
- التحويل بين المستودعات يجب أن يحافظ على نفس المتغير.
- التقرير يجب أن يعرض التجميع حسب المادة واللون والمقاس.
- النصوص الظاهرة لا تعرض مصطلح SKU، وتستخدم الترجمات: رمز المتغير / Variant code / Variantencode.
- وضع API/Network يجب أن يستخدم نفس payload للمتغيرات.
- RTL/LTR والطباعة الموحدة لا يتم تجاوزهما.

## إصلاح مهم

تم تقوية عكس حركات المستودع محليًا وعلى الخادم بحيث يتم التجميع والعكس حسب:

- المادة.
- المستودع.
- نوع الحركة.
- الكلفة.
- معرف المتغير.
- اللون.
- المقاس.
- رمز المتغير.
- نوع الباركود.
- الباركود المطابق.

هذا يمنع الخلط بين:

- قميص / أبيض / M
- قميص / أبيض / L
- قميص / أسود / M
- قميص / أسود / L

عند عكس فاتورة أو مرتجع أو حركة مستودعية.

## نتيجة المرحلة

بعد هذه المرحلة يصبح مسار الألبسة الأول جاهزًا تشغيليًا كمسار متكامل:

`مادة → ألوان/مقاسات → شراء → بيع بالباركود → مرتجع/عكس → تحويل → جرد → تقرير`
